## Scope

PulseVoice is a single-tenant, single-user, local-first macOS daemon that taps Claude Code's structured JSONL transcript and narrates it aloud through an on-device pipeline. The threat model is deliberately narrow because the system has no public surface and no accounts; it exists only on the operator's own machine, reading data the operator already owns.

**Sensitive data handled (phase 4.1.1).** No regulated PII, health, financial, or payment data is processed. The data is sensitive in a different way: it narrates the operator's own source code, architecture, file paths, and agent reasoning, and may *incidentally* capture secrets that appear in agent output — environment variables, tokens embedded in a Bash command, API keys inside a diff. This is the load-bearing sensitivity class for the project. Cloud-TTS exfiltration of any of this is a real concern, which is why local-first Kokoro TTS is the security-driven default (not merely a cost optimization), enforced by NFR-13. Secret-redaction *before speaking* is a recorded aware-but-deferred v1 item under the explicit assumption of a private/trusted audio environment (NFR-16).

**Regulated domain (phase 4.1.2).** None. PulseVoice is a personal-use developer tool and is not subject to GDPR, HIPAA, SOC 2, or PCI-DSS. There are no user accounts, no data collection, and no third parties in the v1 default path.

**External attack surface (phase 4.3.1).** Effectively none — the daemon is local-only. It binds to localhost (loopback / Unix domain socket) with no public internet exposure; v1 single-machine operation is air-gapped-equivalent. The single open nuance is the homelab two-node possibility (spec section 11): a Mac Mini and an ROG laptop where the agent runs on a different machine than the operator. The confirmed stance is that any daemon↔node traffic that crosses the LAN is trusted-LAN-only / authenticated, *never* public.

**Auth model (phase 4.2.1).** None. A single local user on their own machine. Daemon↔menubar IPC is local-only (loopback / Unix socket). The only OS-level grant is the one-time macOS accessibility permission for global hotkeys (FR-2) — an OS capability, not authentication.

**Tenancy model (phase 4.2.2).** Single-tenant, single-user. One operator, one machine. Multi-session multiplexing (several concurrent Claude Code terminals) is handled *inside* the daemon, but the security model is not multi-user.

## STRIDE checklist

Each category is assessed against PulseVoice's actual attack surface, not generically. Categories marked **lower priority** are not dismissed — they are deprioritized because loopback-only IPC, the absence of accounts, and the single-tenant stance remove the precondition that would make them exploitable in the v1 default path. The reasoning is recorded so a future change (a networked deployment, a second user, a cloud path enabled) reopens the category.

| STRIDE category | In scope? | Rationale + Mitigation strategy |
| --- | --- | --- |
| **Spoofing** | **No (lower priority)** | No user accounts and no networked identity exist to spoof. The daemon speaks only to the local operator via loopback / Unix domain socket (NFR-14), and there is no auth layer to defeat because there is nothing to authenticate *to* (NFR-15). A hostile local process *could* in principle connect to the Unix socket, but on a single-user machine the attacker is already the operator; the threat is vacuous in v1. Client-spoofing of Claude Code hooks is bounded by FR-24 (hook fails fast and logs when no daemon listens) and by the schema-version probe (FR-23) which rejects transcript shapes that do not conform. Reopens if a multi-user or networked deployment is ever considered. |
| **Tampering** | **Yes** | The Claude Code JSONL transcript is an *unversioned internal format of a third-party tool that ships changes often* (Phase 5 / Arch-review C5) — the most realistic "the input silently shifted" event in the system. Mitigation is the schema-version probe in the source adapter with **loud** flattened-text degradation (FR-23, NFR-22): a parse failure degrades narration quality and announces itself rather than masquerading as normal operation. The pipeline MUST NOT panic on a malformed or poisoned transcript (NFR-12); a bad segment degrades only that segment/session while the always-on daemon stays alive. Surface-agnosticism (NFR-11) constrains the blast radius — source-specific shapes are mapped to source-neutral `Segment` metadata by the adapter and never trusted beyond it. |
| **Repudiation** | **No (lower priority)** | Repudiation presupposes multiple parties disputing an action; PulseVoice has one operator who is also the only actor. There is no transaction to repudiate. For operational visibility, all hook-event receipts, transcript parse outcomes (including schema-probe degradations), chunk decisions, TTS first-audio latency, and barge-in events are written to structured local logs at `~/Library/Logs/PulseVoice/` with no remote telemetry (NFR-17) — so the operator can reconstruct what the daemon did if a session misbehaved. |
| **Information disclosure** | **Yes (highest priority)** | The defining sensitivity of the project. Agent output can contain source code, file paths, architecture, agent reasoning, and incidental secrets (env vars, tokens, API keys in diffs). Two disclosure vectors are tracked separately: (a) **cloud exfiltration** — mitigated by the local-first default: Kokoro TTS runs on-device and the v1 default path sends nothing to any cloud service (NFR-13); cloud TTS / LLM slots exist only behind opt-in trait seams and are Phase 2. (b) **audio-side disclosure** — speaking sensitive content aloud into a shared space (shoulder-surfing analog: smart speakers, meeting mics, shared offices). This is *not* automatically lower-risk than cloud exfil (Close-audit C2). v1's mitigation is the recorded private/trusted-audio-environment assumption (NFR-16): the operator is alone, on their own speakers/headphones. Secret-redaction before speaking is deferred to Phase 2 and MUST be revisited before any shared or public-space use. |
| **Denial of service** | **Yes** | The daemon is the always-on safety net — it must not die on bad input, because if it dies, a stepped-away operator can sit silently blocked on a permission prompt (the nightmare case from Phase 5). Mitigation is NFR-12 (no-panic resilience on malformed transcript, Kokoro failure, or poisoned segment — degrade the affected segment, keep the daemon alive) plus FR-24 (when a hook fires and no daemon is listening, the hook logs and exits non-zero rather than blocking the agent). The launchd user agent (FR-1) auto-restarts the daemon on crash. The mode toggle doubles as a built-in kill switch so a misbehaving build can be silenced without halting the agent. |
| **Elevation of privilege** | **No (lower priority)** | There is no privilege separation to elevate across: single user, single tenant, no auth (NFR-15). The only OS-level grant is the one-time macOS accessibility permission for global hotkeys (FR-2), which is an OS capability rather than a PulseVoice credential, and is requested by the OS prompt — not something a remote attacker can grant themselves by talking to the daemon. Daemon↔node LAN traffic, if it ever exists, is trusted-LAN-only / authenticated and never public (NFR-14), which is the only place an elevation story could begin. |

## Trust boundaries

Untrusted input crosses into PulseVoice at a small number of concrete entry points, and the system's stance is that **the only genuinely untrusted content is the agent's own output** — everything else runs on a loopback-only, single-user machine that the operator already fully controls.

The primary trust boundary is the **Claude Code JSONL transcript**, read by the `pulse-source` adapter. This is a third-party, unversioned, frequently-changing format (Phase 5 / Arch-review C5) and is treated as untrusted structured data, not as a stable contract. The adapter owns two sanitization responsibilities: first, a **schema-version probe** that detects a changed or unversioned shape and degrades loudly to flattened-text narration with a spoken "degraded mode" notice and a persistent menubar badge (FR-23, NFR-22) rather than silently mis-parsing; second, the **source-neutral metadata mapping** (Phase 3 / Close-audit C5, NFR-11) — the adapter is solely responsible for translating Claude-specific transcript shapes into source-neutral `Segment` fields (role, kind, file path, op type, line count). Everything downstream of `pulse-source` reads only those neutral fields and is compile-time prevented from knowing which agent produced the text, which bounds the blast radius of any transcript-shape tampering to the adapter itself.

The second boundary is the **Unix domain socket** that carries hook events from ephemeral Claude Code hook subprocesses into the resident daemon (Phase 3 / Arch-review C3, Phase 5.1). It is deliberately *not* HTTP and *not* a watched file. The socket is loopback-only (NFR-14): no public exposure. The validation posture on this boundary is fail-safe and non-blocking toward the agent — when a hook fires and no daemon is listening, the hook logs and exits non-zero rather than hanging the agent (FR-24); when the daemon is listening, incoming events feed a pipeline that MUST NOT panic on any input (NFR-12).

The third boundary is the **operator-edited config file** (voice/persona, hotkey bindings, trivial-turn and chunker thresholds, FR-3). It is trusted-by-construction — the operator owns it and edits it deliberately — but it is still parsed by the daemon, so the no-panic discipline (NFR-12) applies: a malformed config degrades to defaults rather than crashing the always-on process.

The fourth, orthogonal boundary is the **macOS accessibility grant** (FR-2). This is not auth and is not PulseVoice-issued; it is an OS capability gated by the standard macOS TCC prompt, requested once, and the daemon functions as a narrator even before it is granted (only hotkey control degrades). There is no trust boundary inside PulseVoice that depends on it.

Finally, the **homelab two-node case** (Phase 4.3.1, spec section 11) would introduce a *new* trust boundary — daemon↔node LAN traffic — that does not exist in the v1 single-machine default. The recorded stance is that such traffic, if it ever ships, MUST be trusted-LAN-only / authenticated and never public (NFR-14); the concrete transport (mTLS, SSH tunnel, WireGuard) is not yet specified and is captured as an open item below.

## Open threat items

These are living items — concrete exposures drawn from Phase 3 and Phase 4 that are not yet fully mitigated in v1. Each should be addressed during the relevant slice of work and linked to its mitigation slice when one lands.

1. **Incidental secret vocalization (audio-side disclosure).** Agent output can contain secrets — env vars, tokens in a Bash command, API keys in a diff — and v1 speaks that output aloud via local Kokoro TTS. Cloud exfiltration is mitigated by the local-first default (NFR-13), but the audio-side vector (smart speakers, meeting mics, shared spaces, shoulder-surfing) is *not* mitigated: speaking a secret aloud is not automatically lower-risk than exfil'ing it (Close-audit C2 / phase 4.1.1). v1's mitigation is the recorded private/trusted-audio-environment assumption (NFR-16), which is an explicit assumption, not a silent claim of safety. **Open action:** revisit a minimal secret-pattern suppressor in Phase 2, and *especially* before any shared or public-space use. Track against the slice that introduces the first non-private audio environment.

2. **Unversioned JSONL transcript durability.** The Claude Code transcript is an unversioned third-party format that ships changes often (Phase 5 / Arch-review C5). The schema-version probe + loud flattened-text degradation (FR-23, NFR-22) is the designed mitigation, but the mitigation itself is a standing risk: if the probe mis-detects a shape change (silently accepting a drifted schema, or noisily degrading on an unrelated field), narration quality silently drops or the operator is flooded with false degraded-mode alerts. **Open action:** validate the probe against real upstream transcript drift as part of the source-adapter slice, and ensure the loud-degradation path is covered by golden tests so it cannot silently regress. Mitigation partial; this is the realistic "it stopped working" event for the project.

3. **Homelab two-node LAN transport is unspecified.** If the two-node setup (Mac Mini ↔ ROG laptop, Phase 4.3.1 / spec section 11) ever ships, daemon↔node traffic would cross a LAN and introduce a real network trust boundary where none exists in v1. The recorded stance — trusted-LAN-only / authenticated, never public (NFR-14) — asserts the *property* but names no *mechanism* (mTLS, SSH tunnel, WireGuard, loopback-over-SSH). **Open action:** before any cross-machine slice begins, pick and record the concrete authenticated transport in an ADR (see ADR 0001's standing note on the trusted-LAN stance), and revisit the Spoofing and Elevation-of-privilege categories above — both reopen the moment traffic leaves loopback.
